## 1) 晋升系统

- **分类**：System Architecture
- **卡片简介**：无专职设计师场景下，以前端负责人兼 UI/产品补位推进 0→1；二期一次性全量上线，缺陷修复及时率 100%，稳定支撑公司晋升评审。
- **卡片标签**：流程治理、跨部门协作、高质量交付
- **结果标签**：0→1、跨部门
- **阶段/时间**：0 → 1 / 2025 Q1-Q2
- **角色定位**：前端主负责人（兼 UI/产品补位）
- **亮点**：
  - 无专职设计师：前端负责人兼 UI/产品补位
  - 二期一次性全量验收通过并当日上线
  - 缺陷修复及时率 100%

### 结果与影响

- **目标**：把线下晋升的低效与不透明，升级为可支撑全公司多序列评审的数字化平台，并确保一、二期按期稳定落地。
- **背景（冲突）**：周期紧、模块多，且无专职 UI；需同时完成交互与视觉统一、核心模块攻坚，以及跨团队并行联调。
- **决策（取舍）**：在“上线速度 vs. 体验完整度”的冲突下，先锁定评审主链路和风险控制点，非关键增强延后到二期，确保一期按时且可稳定运行。
- **行动（你主导）**：我拍板前端交付节奏与分工优先级；亲自承担高难模块（评分、候选人信息、富文本交互）；推动产品、HR 业务方、后端、测试在统一验收口径下协同。
- **关键动作（方法）**：
  - 先梳理评审主链路与规则口径，再统一 10动态交互设计语言，降低评审场景体验分叉与返工。
  - 封装 12 个高复用业务组件（评分器/候选人信息港/富文本/筛选等），并组织 8 个关键模块联调与验收清单对齐。
  - 在 HR 工作台多人协作阶段引入 Rspress 组件文档生成脚本，提交代码时同步更新公共组件文档，治理重复封装与命名漂移。
- **结果（数字）**：一、二期均按期上线（二期一次性全量验收通过并当日上线）；缺陷修复及时率 100%；沉淀组件与协作机制在 HR 相关场景复用，降低重复开发。
- **沉淀（可复用）**：沉淀 12 个高复用组件 + 页面设计语言 + 联调清单 + Rspress 组件文档维护机制，后续在 HR/DSTE 场景复用，减少重复开发与风格漂移。

### 案例与实现

1. 工程侧以评审主链路为中心拆分模块边界：围绕评分、候选人信息港、富文本等高风险交互做组件化抽象，统一状态管理与异常兜底策略，降低联调返工。
2. 建立可复用的页面级交互规范与组件契约（Props/事件/埋点字段），配合走查清单与灰度开关，保障高峰期评审体验与数据一致性。
3. 在 HR 工作台多人协作背景下，落地 Rspress 组件文档生成脚本，让公共组件在提交阶段持续可发现、可检索，减少重复封装与命名漂移。

## 2) 异动系统

- **分类**：Mobile Development
- **卡片简介**：团队与业务方双重变动下 1 个月救场：先对齐口径与上线标准，再落地移动端核心链路，保障业务连续并形成可复用推进模板。
- **卡片标签**：快速接手、移动端重构、复杂协同
- **结果标签**：移动端、复杂协同
- **阶段/时间**：攻坚交付 / 2025 H1
- **角色定位**：项目承接人 + 前端主责
- **亮点**：
  - 团队与业务方双重变动下的 1 个月救场
  - 以产品视角对齐口径后推动主路径上线
  - 移动端 0→1 完成基础搭建并进入迭代

### 结果与影响

- **目标**：在交接期混乱中快速承接异动系统，保障业务连续，并完成移动端 0→1 搭建，避免项目长期悬空。
- **背景（冲突）**：技术、产品、业务侧同时更替，历史口径与优先级不清；短期窗口内既要稳业务又要控范围。
- **决策（取舍）**：在“快速上线 vs. 需求完备”冲突下，优先上线高频链路，低优先需求进入后续迭代，避免继续因范围膨胀导致延期。
- **行动（你主导）**：我拍板“先解沟通卡点、再推上线”的推进策略；亲自完成口径梳理与移动端主路径方案；推动业务方、产品、研发在一个版本窗口内达成上线共识。
- **关键动作（方法）**：
  - 以产品视角先对齐业务口径与上线标准，再反推研发排期；用“最小上线包”优先打通高频核心链路。
  - 持续处理线上问题并统一多端交互，将富文本等通用能力抽象复用到 HR/DSTE 场景，降低重复建设。
- **结果（数字）**：异动系统平稳承接并持续迭代，业务无断档；移动端完成基础搭建并进入后续需求开发；沉淀“卡点诊断→口径对齐→最小上线包”模板便于复用。
- **沉淀（可复用）**：形成“卡点诊断→口径对齐→最小上线包”推进模板，并复用到后续 HR 移动端需求推进中。

### 案例与实现

1. 交接期先完成“口径—范围—验收”三角对齐：把高频链路拆成最小上线包，明确接口契约与回归边界，避免范围膨胀拖垮窗口。
2. 移动端围绕申请、审批、通知做信息架构收敛：统一状态表达与异常兜底，让一线效率与合规要求同时可预期。
3. 上线后持续消化线上反馈，抽象富文本等通用能力沉淀到 HR/DSTE 场景，降低后续需求重复建设成本。

## 3) 版本管理系统

- **分类**：Product Management
- **卡片简介**：作为前端技术负责人搭建多业务线版本/项目统一底座：双端协同、发布治理可审计，首年故障率 < 0.5%。
- **卡片标签**：多端协同、产品化建设、稳定上线
- **结果标签**：产品化、稳定上线
- **阶段/时间**：0 → 1 / 产品化 / 2024-2025 H1
- **角色定位**：前端技术负责人 + 产品化推动者
- **亮点**：
  - DSTE 关键流程数字化覆盖率 90%
  - 与下游数据自动同步
  - 系统首年故障率 < 0.5%

### 结果与影响

- **目标**：打通 DSTE 与 HR 的版本/项目管理割裂，用标准化、可追溯的发布治理支撑双端协同迭代。
- **背景（冲突）**：多业务线流程差异大、权限与数据同步复杂；大版本一次性上线风险高，需要可观测与可回滚。
- **决策（取舍）**：在“功能广度 vs. 系统稳定性”冲突下，采用小步快跑版本化发布（v1.4.0→v1.14.0），以可观测和可回滚优先，避免大版本一次性风险。
- **行动（你主导）**：我拍板版本节奏与能力拆分方式；亲自主导关键模块方案评审与疑难联调；推动研发、测试、运维建立统一发布与回滚协作机制。
- **关键动作（方法）**：
  - 以“小步快跑”版本节奏（v1.4.0→v1.14.0）拆分能力模块，配合灰度、回滚与监控机制降低一次性大版本风险。
  - 沉淀组件库/脚手架与规范，组织双周会与月刊推进协同，并落地 AI 提效实践缩短交付周期。
- **结果（数字）**：DSTE 关键流程数字化覆盖率达 90%，与下游数据自动同步；前端 AI 提效约 5%-10%，系统首年故障率 < 0.5%。
- **沉淀（可复用）**：沉淀版本发布节奏、组件规范、双周会/月刊机制和 AI 提效实践，可迁移到新系统 0→1 与存量系统治理。

### 案例与实现

1. 用统一版本模型承载规划、节奏、变更与回滚策略，并把“发布节点—责任人—验收口径”固化成可执行流程，减少跨团队口头对齐。
2. 双端采用“统一能力底座 + 差异化展示”，在权限同步与数据一致性上优先保障关键路径可观测、可回滚。
3. 以双周会/月刊机制推进协同，同时把组件库、脚手架与规范落到日常交付，让迭代节奏可预测、风险可前置。

## 4) 飞块低代码平台 3.0

- **分类**：Low-Code Platform
- **卡片简介**：Owner 全闭环主导自研低代码平台：模块化 + 可视化配置落地，结合推广与模板化赋能，规模化支撑业务搭建并显著降本。
- **卡片标签**：低代码、平台架构、MVP 规划
- **结果标签**：低代码、平台化、降本增效
- **阶段/时间**：低代码创新 / 平台化 / 2023-2025
- **角色定位**：低代码平台前端 Owner（方案、架构、推广）
- **亮点**：
  - 稳定运行 16 个月
  - 300+ 应用，1000+ 页面
  - 保守估计节省约 400w 成本

### 结果与影响

- **目标**：用自研低代码平台替代外采方案，在信息安全与定制性约束下，提升页面搭建效率并降低长期成本。
- **背景（冲突）**：既有低代码能力难以覆盖复杂场景；外部平台定制性与信息安全约束明显，且长期成本较高。
- **决策（取舍）**：在“平台通用性 vs. 场景定制”冲突下，先抽象通用能力并保留扩展插槽，避免早期过度定制造成后续维护成本激增。
- **行动（你主导）**：我拍板 MVP 边界与核心架构路线；亲自完成关键技术验证与前端方案落地；推动业务团队从“提需求”转为“可自主搭建”协作模式。
- **关键动作（方法）**：
  - 完成竞品与差距分析，定义 MVP 与“模块化 + 可视化配置”架构，并完成关键技术验证与前端方案落地。
  - 编写指引/案例模板并组织宣讲推广，推动业务从“提需求开发”转向“可自主搭建”，以组件化提升一致性。
- **结果（数字）**：平台稳定运行并支撑规模化使用：单页面开发耗时从“按天”降至“按小时”；结合规模与人力折算，保守估计节省约 400w 成本。
- **沉淀（可复用）**：形成低代码平台的能力分层、模板化搭建与推广文档体系，可复用于后续平台版本与跨团队复制。

### 案例与实现

1. 从竞品与外采约束反推 MVP：用“通用能力 + 扩展插槽”平衡定制与维护成本，并完成关键技术验证与前端架构落地。
2. 把搭建体验产品化：沉淀指引、案例模板与组件标准，让业务同学能自助搭建复杂页面，同时保持风格与交互一致。
3. 通过宣讲、答疑与持续迭代建立推广闭环，把平台能力从“能用”推进到“多团队日常深度使用”。

## 5) 价值投资项目

- **分类**：Multi-Agent Architecture
- **卡片简介**：价值投资场景多智能体投研：借鉴 Claude / Claude Code 做意图识别与简单/复杂分流，ReAct 与 Plan-ReSolve 调研选型后落地多 Agent 编排（主 Agent 汇总、质疑校准）+ SSE 可观测；PC+iOS 覆盖分析—任务—复盘闭环。
- **卡片标签**：多Agent、PC+iOS 双端、SSE 流式、任务编排
- **结果标签**：多Agent、双端交付、任务编排、可观测
- **阶段/时间**：策略项目 / 多Agent落地 / 2025
- **角色定位**：产品经理 + 技术方案推动者
- **亮点**：
  - 多智能体编排 + SSE 可观测
  - PC+iOS 双端一致体验
  - Plan-and-Execute 任务流可复盘

### 结果与影响

- **目标**：把投研里“长链路、难协同、难复盘”的分析型工作，产品化为可规划、可执行、可观测的多智能体任务系统，并覆盖 PC 与 iOS。
- **背景（冲突）**：传统流程依赖人工串联，复杂问题缺少任务化承载；长链路输出波动大，移动端难以及时跟进过程与结论。
- **决策（取舍）**：在“自动化速度 vs. 输出可靠性”冲突下，先保证可解释与可追溯（SSE 过程可见 + 状态落库 + 人机审核节点），再逐步提高自动化比例。
- **行动（你主导）**：我拍板多 Agent 能力拆分、任务状态机与双端交付节奏；亲自定义关键交互与事件映射；推动后端、前端、业务在统一任务语义与输出格式下协同。
- **关键动作（方法）**：
  - 借鉴 Claude / Claude Code「意图优先、过程可感」的思路：对用户做意图识别，区分简单任务与复杂长链路任务；调研 ReAct 与 Plan-ReSolve 等机制后，结合可解释、可审计目标判断与系统更契合的编排方式；落地多 Agent 编排（主 Agent 汇总结论、质疑/校准 Agent 纠偏），入口只做协议与校验，并用统一 SSE 表达推理、工具与分段输出。
  - 落地 Plan-and-Execute + 状态机：规划确认、分步执行、结果落库，并接入调度支持周期再跑，保证过程可解释与可审计。
  - 主导 PC/iOS 双端：统一信息架构与状态模型，覆盖任务列表、执行进度、结果详情与复盘视图，降低跨端协作成本。
- **结果（数字）**：沉淀可复用的多智能体投研底座：把复杂分析从“人肉推进”转为“可规划、可执行、可观测”；双端协同后跨角色对齐与复盘效率显著提升。
- **沉淀（可复用）**：沉淀意图分层策略、多 Agent 角色编排（主 Agent / 质疑校准）、任务状态机、SSE 事件规范与双端页面框架，可迁移到其他分析型与任务型业务场景。

### 案例与实现

1. 产品与技术方案借鉴 Claude / Claude Code：意图识别区分简单与复杂任务；调研 ReAct 与 Plan-ReSolve 后选型，落地多 Agent 编排（主 Agent 汇总、质疑 Agent 校准）。后端以 FastAPI 承载：统一入口按意图分流到股票分析、Mission、模型工坊等，SSE 输出推理、工具与分段结果，降低黑盒感。
2. 长任务用 Plan-and-Execute + 状态机固化“规划确认 → 分步执行 → 落库沉淀”，并支持周期再跑，让过程可解释、可审计、可复盘。
3. 前端主导 PC/iOS：抽象任务状态与事件映射，统一信息架构与关键页面（列表/进度/结果/复盘），让移动场景也能跟进过程与结论。

### 5) 价值投资项目（详情）

- **时间**：2025

- **背景（冲突）**：传统流程依赖人工串联，复杂问题缺少任务化承载；长链路输出波动大，移动端难以及时跟进过程与结论。
- **决策（取舍）**：在“自动化速度 vs. 输出可靠性”冲突下，先保证可解释与可追溯（SSE 过程可见 + 状态落库 + 人机审核节点），再逐步提高自动化比例。
- **行动（你主导）**：我拍板多 Agent 能力拆分、任务状态机与双端交付节奏；亲自定义关键交互与事件映射；推动后端、前端、业务在统一任务语义与输出格式下协同。
- **关键动作（方法）**：
  1. 借鉴 Claude / Claude Code「意图优先、过程可感」的思路：对用户做意图识别，区分简单任务与复杂长链路任务；调研 ReAct 与 Plan-ReSolve 等机制后，结合可解释、可审计目标判断与系统更契合的编排方式；落地多 Agent 编排（主 Agent 汇总结论、质疑/校准 Agent 纠偏），入口只做协议与校验，并用统一 SSE 表达推理、工具与分段输出。
  2. 落地 Plan-and-Execute + 状态机：规划确认、分步执行、结果落库，并接入调度支持周期再跑，保证过程可解释与可审计。
  3. 主导 PC/iOS 双端：统一信息架构与状态模型，覆盖任务列表、执行进度、结果详情与复盘视图，降低跨端协作成本。
- **结果（数字）**：沉淀可复用的多智能体投研底座：把复杂分析从“人肉推进”转为“可规划、可执行、可观测”；双端协同后跨角色对齐与复盘效率显著提升。
- **沉淀（可复用）**：沉淀意图分层策略、多 Agent 角色编排（主 Agent / 质疑校准）、任务状态机、SSE 事件规范与双端页面框架，可迁移到其他分析型与任务型业务场景。
- **案例与实现**：
  1. 产品与技术方案借鉴 Claude / Claude Code：意图识别区分简单与复杂任务；调研 ReAct 与 Plan-ReSolve 后选型，落地多 Agent 编排（主 Agent 汇总、质疑 Agent 校准）。后端以 FastAPI 承载：统一入口按意图分流到股票分析、Mission、模型工坊等，SSE 输出推理、工具与分段结果，降低黑盒感。
  2. 长任务用 Plan-and-Execute + 状态机固化“规划确认 → 分步执行 → 落库沉淀”，并支持周期再跑，让过程可解释、可审计、可复盘。
  3. 前端主导 PC/iOS：抽象任务状态与事件映射，统一信息架构与关键页面（列表/进度/结果/复盘），让移动场景也能跟进过程与结论。
