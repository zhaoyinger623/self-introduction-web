## 1) 晋升系统

- **分类**：System Architecture
- **卡片简介**：在晋升系统从0到1的全链路建设中，以“技术攻坚 + 跨角色补位 + 质量交付”三重能力突破项目难点。
- **卡片标签**：流程治理、跨部门协作、高质量交付
- **结果标签**：0→1、跨部门
- **阶段/时间**：0 → 1 / 2025 Q1-Q2
- **角色定位**：前端主负责人（兼 UI/产品补位）
- **亮点**：
  - 获评公司历史体验最佳晋升系统
  - 二期一次性全量验收通过并当日上线
  - 缺陷修复及时率 100%

### 结果与影响

- **目标**：完成晋升系统数字化平台前端侧从 0 到 1 建设，确保一期/二期按期上线并可稳定支撑评审流程。
- **背景（冲突）**：时间紧、任务重，且团队无专职 UI 设计师；需同步承担方案对齐、UI 统一和多模块并行联调。
- **决策（取舍）**：在“上线速度 vs. 体验完整度”的冲突下，先锁定评审主链路和风险控制点，非关键增强延后到二期，确保一期按时且可稳定运行。
- **行动（你主导）**：我拍板前端交付节奏与分工优先级；亲自承担高难模块（评分、候选人信息、富文本交互）；推动产品、HR 业务方、后端、测试在统一验收口径下协同。
- **关键动作（方法）**：
  - 统一 5 个核心页面与动态交互设计语言后再并行开发，提前消除评审链路的体验分叉风险。
  - 完成 12 个高复用组件封装（评分器/候选人信息港/富文本/筛选等），并组织 8 个关键模块联调。
- **结果（数字）**：一期、二期均按期上线（含二期一次性全量验收通过并当日上线）；缺陷修复及时率 100%，上线效果获业务方好评。
- **沉淀（可复用）**：沉淀 12 个高复用组件 + 页面设计语言 + 联调清单，后续在 HR/DSTE 场景复用，减少重复开发与风格漂移。

### 案例与实现

1. 主导晋升系统从 0 到 1 的建设，覆盖需求澄清、流程建模、评审规则配置与结果落地，在复杂组织场景下保证流程可解释、可追溯。  
2. 在关键节点承担技术攻坚与跨角色沟通，与业务、HR、法务等多方对齐口径，推动方案在约束条件下按期上线。  
3. 通过评审看板、权限与审计设计，降低操作风险；配合联调与灰度策略，保障高峰期评审体验与数据一致性。

## 2) 异动系统

- **分类**：Mobile Development
- **卡片简介**：在技术团队、产品经理、业务方均发生人员更换的艰难处境下，一个月内梳理已有逻辑，落地移动端系统开发。
- **卡片标签**：快速接手、移动端重构、复杂协同
- **结果标签**：移动端、复杂协同
- **阶段/时间**：攻坚交付 / 2025 H1
- **角色定位**：项目承接人 + 前端主责
- **亮点**：
  - 业务沟通卡点 3 个月未上线
  - 转为产品经理视角介入后 1 个月推动上线
  - 移动端从 0 到 1 进入开发阶段

### 结果与影响

- **目标**：在人员更替和需求变化下快速承接异动系统，确保业务连续运转并推进移动端从 0 到 1 搭建。
- **背景（冲突）**：技术团队、产品经理、业务侧均发生变动，需要在短周期内重建需求认知与交付节奏。
- **决策（取舍）**：在“快速上线 vs. 需求完备”冲突下，优先上线高频链路，低优先需求进入后续迭代，避免继续因范围膨胀导致延期。
- **行动（你主导）**：我拍板“先解沟通卡点、再推上线”的推进策略；亲自完成口径梳理与移动端主路径方案；推动业务方、产品、研发在一个版本窗口内达成上线共识。
- **关键动作（方法）**：
  - 用产品经理视角先对齐业务口径与上线标准，再反推研发排期，1 个月内打通上线主路径。
  - 持续处理线上问题、推动样式与交互统一，并将富文本等通用能力抽象复用到 HR/DSTE 场景。
- **结果（数字）**：完成异动系统平稳承接并持续迭代，保障业务无断档；移动端项目完成基础搭建并进入需求开发阶段。
- **沉淀（可复用）**：形成“卡点诊断→口径对齐→最小上线包”推进模板，并复用到后续 HR 移动端需求推进中。

### 案例与实现

1. 在团队与业务方人员更替的背景下，快速梳理历史逻辑与数据口径，输出可执行的迭代计划，在约一个月内完成移动端核心链路交付。  
2. 针对异动申请、审批与通知等场景做信息架构与交互收敛，兼顾一线使用效率与管理合规要求。  
3. 与后端、测试协同推进接口契约与回归范围，在有限时间内控制变更风险，保障上线后业务可连续运转。

## 3) 版本管理系统

- **分类**：Product Management
- **卡片简介**：设计并实现了多端适配的版本管理系统，支持Web、移动端等多种平台。
- **卡片标签**：多端协同、产品化建设、稳定上线
- **结果标签**：产品化、稳定上线
- **阶段/时间**：0 → 1 / 产品化 / 2024-2025 H1
- **角色定位**：前端技术负责人 + 产品化推动者
- **亮点**：
  - DSTE 关键流程数字化覆盖率 90%
  - 与下游数据自动同步
  - 系统首年故障率 < 0.5%

### 结果与影响

- **目标**：建立可规模化复用的版本/项目管理能力，支撑多业务线数字化流转与发布治理。
- **背景（冲突）**：需兼顾 DSTE 与 HR 双工作台演进，处理多流程配置、权限同步、可追溯与一致性。
- **决策（取舍）**：在“功能广度 vs. 系统稳定性”冲突下，采用小步快跑版本化发布（v1.4.0→v1.14.0），以可观测和可回滚优先，避免大版本一次性风险。
- **行动（你主导）**：我拍板版本节奏与能力拆分方式；亲自主导关键模块方案评审与疑难联调；推动研发、测试、运维建立统一发布与回滚协作机制。
- **关键动作（方法）**：
  - 按版本节奏驱动能力演进（v1.4.0→v1.14.0），将核心能力拆成可持续上线模块，降低大版本一次性风险。
  - 沉淀组件库/脚手架与规范，组织双周会与月刊，推进 AI 提效方案落地并建立团队协作机制。
- **结果（数字）**：DSTE 关键流程数字化覆盖率达 90%，与下游数据自动同步；前端 AI 提效约 5%-10%，系统首年故障率 < 0.5%。
- **沉淀（可复用）**：沉淀版本发布节奏、组件规范、双周会/月刊机制和 AI 提效实践，可迁移到新系统 0→1 与存量系统治理。

### 案例与实现

1. 从业务目标出发设计版本管理模型，覆盖版本规划、发布节奏、变更记录与回滚策略，支撑多端产品协同迭代。  
2. 实现 Web 与移动端的统一能力底座与差异化展示，在复杂权限与数据同步场景下保持体验一致、操作可预期。  
3. 推动与研发、测试、运维的流程衔接，通过可视化与通知机制降低沟通成本，支撑稳定、可审计的发布闭环。

## 4) 飞块低代码平台 3.0

- **分类**：Low-Code Platform
- **卡片简介**：在平台能力薄弱与外采受限背景下，主导飞块 3.0 方案设计与前端架构升级，推动低代码能力从“可用”走向“可规模化复用”。
- **卡片标签**：低代码、平台架构、MVP 规划
- **结果标签**：低代码、平台化、降本增效
- **阶段/时间**：低代码创新 / 平台化 / 2023-2025
- **角色定位**：低代码平台前端 Owner（方案、架构、推广）
- **亮点**：
  - 稳定运行 16 个月
  - 300+ 应用，1000+ 页面
  - 保守估计节省约 200w 成本

### 结果与影响

- **目标**：在公司低代码能力不足的背景下，构建自有可扩展平台，提升业务搭建效率并降低外采依赖。
- **背景（冲突）**：既有低代码能力难以覆盖复杂场景；外部平台定制性与信息安全约束明显，且长期成本较高。
- **决策（取舍）**：在“平台通用性 vs. 场景定制”冲突下，先抽象通用能力并保留扩展插槽，避免早期过度定制造成后续维护成本激增。
- **行动（你主导）**：我拍板 MVP 边界与核心架构路线；亲自完成关键技术验证与前端方案落地；推动业务团队从“提需求”转为“可自主搭建”协作模式。
- **关键动作（方法）**：
  - 先完成竞品与现状差距分析，再定义 MVP 与“模块化 + 可视化配置”架构，确保首版可落地、后续可扩展。
  - 编写平台指引与案例模板，推动业务侧自主搭建；并以组件化方式提升页面复用与一致性。
- **结果（数字）**：飞块 3.0 平台上线后持续稳定运行，单页面开发耗时从“按天”降至“按小时”，支撑多团队日常业务深度使用。
- **沉淀（可复用）**：形成低代码平台的能力分层、模板化搭建与推广文档体系，可复用于后续平台版本与跨团队复制。

### 案例与实现

1. 基于业务和用户视角完成低代码产品调研，结合飞块 2.0 的能力边界，规划 MVP 并提出模块化 + 可视化配置架构。  
2. 主导平台关键能力设计与前端实现，兼顾可扩展性、组件标准化和搭建体验，确保复杂场景可落地。  
3. 通过指引文档、模板案例与推广支持，降低上手门槛，推动多业务团队自主搭建并稳定使用。

## 5) 价值投资项目

- **分类**：Multi-Agent Architecture
- **卡片简介**：围绕价值投资打造多 Agent 投研系统，前端同步落地 PC 与 iOS 双端，打通“分析—任务—复盘”闭环。
- **卡片标签**：多Agent、PC+iOS 双端、SSE 流式、任务编排
- **结果标签**：多Agent、双端交付、任务编排、可观测
- **阶段/时间**：策略项目 / 多Agent落地 / 2025
- **角色定位**：产品经理 + 技术方案推动者
- **亮点**：
  - 后端多智能体编排
  - PC+iOS 双端落地
  - 任务流可复盘可调度

### 结果与影响

- **目标**：构建价值投资场景下可复用的多智能体投研系统，统一“分析—任务—复盘”闭环，并在 PC 与 iOS 双端可用。
- **背景（冲突）**：传统投研流程依赖人工串联，复杂问题缺少可执行任务化能力；长链路输出不稳定，移动场景难以实时跟进进度与结论。
- **决策（取舍）**：在“自动化速度 vs. 输出可靠性”冲突下，先保证可解释与可追溯（SSE 过程可见 + 状态落库 + 人机审核节点），再逐步提高自动化比例。
- **行动（你主导）**：我拍板多 Agent 能力拆分、任务状态机与双端交付节奏；亲自定义关键交互与事件映射；推动后端、前端、业务在统一任务语义与输出格式下协同。
- **关键动作（方法）**：
  - 以“路由薄、领域厚”思路拆分系统：路由层负责协议与校验，Agent 层负责意图路由、任务编排与多角色协作，并统一 SSE 事件协议。
  - 落地 Mission 的 Plan-and-Execute 与状态机，把任务拆为“规划确认 + 分步执行 + 结果落库”，并接入调度器支持周期再跑。
  - 主导前端 PC 与 iOS 双端实现统一体验：抽象通用组件与状态模型，覆盖任务列表、执行进度、结果详情与复盘视图。
- **结果（数字）**：形成可复用的多智能体投研底座，复杂分析任务由“人肉推进”转为“可规划、可执行、可观测”；双端协同上线后，跨角色沟通与复盘效率显著提升。
- **沉淀（可复用）**：沉淀意图路由模板、任务状态机、SSE 事件规范与双端页面框架，可迁移到其他分析型与任务型业务场景。

### 案例与实现

1. 后端基于 FastAPI 构建多智能体编排：以统一对话入口承接请求，通过意图路由分流到股票分析、Mission 任务流、模型工坊等链路，并用 SSE 持续输出推理、工具调用与结果片段。  
2. 面向长任务场景，引入 Plan-and-Execute 与状态机，把“创建任务→规划→确认→执行→落库→定时再跑”标准化，保证过程可解释、可复盘、可调度。  
3. 前端侧由我主导完成 PC 与 iOS 双端实现：统一信息架构与交互语言，做事件流可视化与任务状态呈现，让业务方在移动场景也能及时查看进度、结论与复盘产物。
